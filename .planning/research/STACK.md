# Technology Stack: Android Mobile Port

**Project:** Spytfy v2.0 Android
**Researched:** 2026-05-22

## Critical Finding: Process Spawning is Blocked

Tauri 2's shell plugin explicitly does NOT support process spawning on Android -- it only allows `open` (URL launching). This means the desktop approach of calling `tokio::process::Command::new("yt-dlp")` will not work through Tauri's plugin system on mobile.

However, `std::process::Command` and `tokio::process::Command` CAN work at the Rust level on Android if the binary is in an executable location (the app's nativeLibraryDir). The limitation is in Tauri's shell plugin abstraction, not in the Android kernel. This opens two viable paths.

**Source:** https://v2.tauri.app/plugin/shell/ (Platform support table: Android = "Only allows to open URLs via `open`")

## Two Viable Approaches

### Approach A: youtubedl-android (Recommended for v2.0)

Use the battle-tested `youtubedl-android` Kotlin/Java library which bundles Python 3.8 + yt-dlp + ffmpeg as native libraries inside the APK. Call it from Rust via a custom Tauri Kotlin plugin.

**Pros:**
- Battle-tested on Android (1800+ GitHub stars, active maintenance, v0.18.1)
- Handles all W^X/SELinux complexity internally (uses nativeLibraryDir with extractNativeLibs=true)
- yt-dlp updates can be pulled dynamically at runtime
- Same yt-dlp behavior as desktop -- scoring algorithm, search results, download flags all transfer directly
- ffmpeg included for audio extraction/transcoding

**Cons:**
- APK size increase (~50-80MB for Python + yt-dlp + ffmpeg)
- Requires Kotlin bridge plugin (Rust -> JNI -> Kotlin -> youtubedl-android -> results back)
- Python 3.8 bundled (old, but sufficient for yt-dlp)
- Cold start is slower (Python initialization)

### Approach B: Rust-Native Pipeline (Higher risk, smaller APK)

Replace yt-dlp entirely with `rusty_ytdl` (pure Rust YouTube client) and ffmpeg with `symphonia` + `mp3lame-encoder`.

**Pros:**
- No external binaries needed -- everything compiles into the Rust .so
- Smaller APK (just Rust code, no Python/ffmpeg binaries)
- Faster execution (no Python interpreter overhead)
- Single language (Rust) for entire backend

**Cons:**
- rusty_ytdl (v0.7.4) has 47K downloads total -- far less battle-tested than yt-dlp
- Last release Aug 2024 (~1.5 years ago) -- maintenance concerns
- YouTube frequently changes its API; yt-dlp has 1800+ extractors and rapid updates, rusty_ytdl does not
- Symphonia does NOT support Opus decoding (marked "-" in v0.6.0 support matrix) -- requires separate Opus decoder
- Multi-crate audio pipeline (rusty_ytdl + symphonia + audiopus + mp3lame-encoder) is unproven as a combo
- If YouTube breaks rusty_ytdl, fixing is slow (small maintainer team vs yt-dlp's massive community)

### Verdict: Approach A for v2.0, Approach B as future optimization

youtubedl-android is the pragmatic choice because:
1. It preserves behavioral parity with the desktop app (same yt-dlp, same search results, same download quality)
2. The Kotlin bridge complexity is manageable (Tauri has first-class Kotlin plugin support)
3. APK size (~80MB) is acceptable for a music downloader
4. The risk profile is dramatically lower than assembling 4 unproven Rust crates

Keep Approach B as a v3.0 goal if APK size or startup speed becomes a user complaint.

## Recommended Stack

### Tauri 2 Android Build

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| Tauri 2 | 2.x (latest stable) | Android app framework | Already used for desktop; Android target supported |
| Android SDK | API 31+ (Android 12) | Target platform | Scoped storage built-in, 65%+ market coverage |
| Android NDK | Latest via SDK Manager | Rust cross-compilation | Required for compiling Rust to aarch64-linux-android |
| JDK | Android Studio bundled JBR | Gradle builds | Use Android Studio's JBR, not system JDK |
| Gradle | Auto-managed by Tauri | Android build system | Generated by `cargo tauri android init` |
| Kotlin | 1.9+ | Custom Tauri plugins | Tauri mobile plugins use Kotlin with @TauriPlugin annotation |

**Confidence:** HIGH -- Tauri 2 Android build is well-documented and stable.

**Rust targets to add:**
```
rustup target add aarch64-linux-android armv7-linux-androideabi i686-linux-android x86_64-linux-android
```

### Android Storage

| Technology | Version | Purpose | Why |
|------------|---------|---------|-----|
| MediaStore API (via custom Kotlin plugin) | Android SDK built-in | Write MP3 files to Music/ directory | On API 31+ apps can insert audio files via MediaStore.Audio.Media without any storage permissions. Uses ContentResolver.insert() + openFileDescriptor() |
| tauri-plugin-fs | 2.x | App-private file access | For SQLite DB, temp files, cover art cache. Works on Android for app-specific dirs |

**Confidence:** HIGH -- MediaStore API is the official Android approach and well-documented.

**No storage permissions needed on API 31+** for writing to Music/ via MediaStore. The app creates files it owns. Key implementation pattern:
1. Write MP3 to app-specific temp directory first (via Rust/tauri-plugin-fs)
2. Copy to Music/ via MediaStore ContentResolver in Kotlin plugin
3. Set IS_PENDING=1 during write, IS_PENDING=0 when complete
4. Use RELATIVE_PATH to organize: `Music/Spytfy/{album}/`

**Confidence:** MEDIUM -- Requires custom Kotlin plugin. No existing Tauri plugin for foreground services.

## Alternatives Considered

| Category | Recommended | Alternative | Why Not |
|----------|-------------|-------------|---------|
| Download engine | youtubedl-android (Kotlin lib) | Sidecar ARM binaries | Tauri shell plugin does not support process spawning on Android. SELinux blocks execution from app data dirs |
| Download engine | youtubedl-android | rusty_ytdl (Rust-native) | Lower maturity (47K downloads vs millions), YouTube breakage risk, Opus decoding gap, maintenance concerns (last release Aug 2024). Consider for v3.0 |
| Download engine | youtubedl-android | Manual ARM yt-dlp in nativeLibraryDir | Could work but requires managing Python+yt-dlp packaging manually. youtubedl-android does this for you |
| Audio transcode | youtubedl-android ffmpeg module | Rust-native (symphonia+mp3lame) | Symphonia lacks Opus decoder. Multi-crate pipeline unproven. Too risky for v2.0 |
| Storage | MediaStore (custom plugin) | MANAGE_EXTERNAL_STORAGE | Google Play rejects for most app categories |
| Storage | MediaStore (custom plugin) | SAF file picker | Poor UX -- user must pick directory every time |
| Storage | MediaStore (custom plugin) | App-specific directory only | Files invisible to other music apps |
| Background | Foreground Service (custom plugin) | WorkManager | WorkManager is for deferrable work. Downloads should happen immediately while user watches |
| TLS | rustls-tls (already in stack) | native-tls / OpenSSL | OpenSSL requires Android NDK linking complexity. rustls is pure Rust |
| Database | SQLx with `bundled` feature | rusqlite | Already using SQLx. No reason to switch |

## What NOT to Add

| Technology | Why Not |
|------------|---------|
| yt-dlp ARM binary (manual sidecar) | youtubedl-android handles this automatically |
| ffmpeg ARM binary (manual sidecar) | youtubedl-android includes ffmpeg module |
| tauri-plugin-shell (for spawning) | Does not support process spawning on Android |
| FFmpegKit | Redundant with youtubedl-android's ffmpeg module |
| MANAGE_EXTERNAL_STORAGE permission | Play Store rejection risk |
| ExoPlayer | Spytfy is a downloader, not a player |
| rusty_ytdl (for v2.0) | Defer to v3.0 as APK size optimization |

```powershell

cd spytfy/src-tauri
cargo tauri android init

# 5. Dev build
cargo tauri android dev

# 6. Release build (generates APK)
cargo tauri android build
```

## Architecture: Desktop vs Android Code Split

```
src-tauri/src/
  download/
    mod.rs          # Shared types (DownloadResult, etc.)
    pipeline.rs     # #[cfg] splits to desktop_pipeline or android_pipeline
    scorer.rs       # SHARED -- pure Rust scoring algorithm
    tagger.rs       # SHARED -- id3 tagging
    cover.rs        # SHARED -- cover art fetching
    verifier.rs     # SHARED -- MP3 verification
    youtube.rs      # DESKTOP ONLY -- yt-dlp subprocess calls
    downloader.rs   # DESKTOP ONLY -- yt-dlp download subprocess
    android.rs      # ANDROID ONLY -- calls Kotlin plugin via Tauri command
```

The Kotlin plugin handles:
- Initializing youtubedl-android
- Running YouTube search (returns JSON with video IDs, titles, durations, channels)
- Running download+transcode (yt-dlp -x --audio-format mp3)
- Writing final MP3 to MediaStore Music/ directory

Rust handles:
- Spotify URL resolution (shared)
- Scoring algorithm (shared)
- ID3 tagging (shared, after Kotlin plugin writes the MP3)
- Cover art (shared)
- Queue management (shared)
- SQLite persistence (shared)

## Future: Rust-Native Pipeline (v3.0)

If APK size or Python startup time becomes a user issue, evaluate:

| Crate | Version | Purpose | Risk |
|-------|---------|---------|------|
| rusty_ytdl | 0.7.4+ | YouTube search + audio download | YouTube breakage, low maintenance velocity |
| symphonia | 0.6.0+ | WebM/MP4 demuxing | Opus decoder still missing as of v0.6.0 |
| audiopus | 0.3+ | Opus decoding (FFI to libopus) | Needs NDK CC for cross-compile |
| mp3lame-encoder | 0.2.4+ | MP3 encoding | Needs NDK CC for cross-compile (LAME C code) |

Audio pipeline: `rusty_ytdl -> symphonia (demux) -> audiopus (decode Opus) -> mp3lame-encoder (encode MP3)`

Fallback: Download M4A/AAC stream (itag 140, 128kbps) instead of WebM/Opus, then `symphonia (demux+decode AAC) -> mp3lame-encoder`. This avoids the Opus gap entirely.
